[![Daily Concert Scrape](https://github.com/Cartograf666/concert-for-travelers-api/actions/workflows/daily-scrape.yml/badge.svg)](https://github.com/Cartograf666/concert-for-travelers-api/actions/workflows/daily-scrape.yml)

[![Static API](https://img.shields.io/badge/API-static%20JSON-blue)](https://cartograf666.github.io/concert-for-travelers-api/)

> Полностью автономный, самовосстанавливающийся пайплайн, который собирает концерты
> с более чем 90 площадок, Ticketmaster, Bandsintown и Eventbrite; сверяет их со
> списком одобренных артистов (whitelist) из более чем 63 000 имён; обогащает
> метаданные артистов через бесплатные структурированные источники и Gemini; и
> публикует всё это в виде бесплатного статического JSON API. Ноль серверов,
> ноль ручного вмешательства в штатном режиме.

Этот репозиторий — только **backend/данные**, фронтенда здесь нет. Отдельное
приложение, [`concerts-for-travelers`](https://github.com/Cartograf666/concerts-for-travelers),
потребляет JSON, публикуемый здесь, чтобы ответить на один вопрос: *«Какие из
артистов, которых я люблю, выступают там, где буду я, и когда?»*

## Источники данных

| Источник | Покрытие | Периодичность | Примечания |
|---|---|---|---|
| **Скрейперы площадок (venue scrapers)** | 91 конфигурация, по одной на площадку | Ежедневно | `static_selectors`, `json_api`, `custom_js`, `jsonld`, `next_data`, `playwright_render` — в зависимости от того, что подходит сайту; самовосстановление поддерживает их в рабочем состоянии. |
| **Ticketmaster Discovery API** | 27 стран | Ежедневно | Широкий дополняющий проход по данным; также выявляет фестивали (события с несколькими исполнителями). |
| **Bandsintown** | По всему миру, по артистам | Ежедневно, пакетами | Единственный источник с реальным охватом рынков, которые упускают остальные (Япония, Россия, ...). Публичный виджет-фид — вежливый темп запросов, с кэшированием, самоограничивается при любых признаках блокировки. |
| **Eventbrite** | США, по артистам | Ежедневно, пакетами | Скрейпинг публичной страницы обнаружения (стороннего API поиска событий больше не существует) — фильтр релевантности по началу имени убирает шум полнотекстового поиска. Компромисс с риском нарушения условий использования (ToS), на который здесь идут, описан в [BACKLOG.md](BACKLOG.md). |
| **Скрейперы тур-страниц артистов** | сейчас 5 конфигураций | Еженедельно | Официальные тур-страницы конкретных артистов — самый надёжный источник по артистам, добавляется вручную. |
| **`discover-artists`** | чарты Deezer + Last.fm | Еженедельно | Самопополняющийся список целевых артистов: подтягивает актуальные глобальные + погенровые/пострановые чарты, чтобы отслеживаемые артисты не отставали от реальной популярности — без участия человека в курации. |

Каждое исходное событие — независимо от источника — проходит через один и тот же
[whitelist одобренных артистов](#key-features), нормализацию дат и пайплайн
дедупликации перед публикацией.

## Ключевые возможности

1. **Архитектура с нулевым обслуживанием, полностью автономная (Zero-Ops)** — работает целиком внутри GitHub Actions и разворачивается статически на GitHub Pages. Ни базы данных, ни сервера, ни ручного вмешательства в штатном режиме: скрейпинг, обнаружение, обогащение, самовосстановление и публикация — всё это выполняется и самокорректируется по расписанию cron.
2. **Многоуровневый каскад обогащения артистов «сначала бесплатное»** — website и соцсети (Spotify/Instagram/Facebook/YouTube/Telegram/VK) заполняются в порядке возрастания стоимости: сначала массовый проход Wikidata SPARQL (~80 имён за запрос), затем поартистный проход MusicBrainz + Wikidata, и только потом Gemini — для оставшегося длинного хвоста. Благодаря этому подавляющее большинство whitelist из ~63 000 артистов обогащается без единой траты на LLM.
3. **Отказоустойчивое переключение между несколькими ключами Gemini (Multi-Key Failover)** — и обогащение, и самовосстановление перебирают все настроенные ключи (`GEMINI_API_KEY`, `_2.._10`, `_RESERV1..10`, либо массовый список `GEMINI_API_KEYS`) и каскад моделей (уровни Gemma → Gemini Flash), переходя к следующему ключу только после того, как на текущем исчерпаны квота/авторизация у всех моделей — это умножает суточный бюджет бесплатного тарифа без каких-либо изменений в коде, кроме добавления секрета.
4. **Многоуровневое сопоставление с whitelist артистов** — точное совпадение → фильтр кавер/трибьют-групп по целым словам → сопоставление по подстроке с защитой по покрытию (с проверкой на присоединённого соседа с заглавной буквы, чтобы короткая запись не могла поглотить фрагмент более длинного, неодобренного имени) → нечёткое сопоставление по редакционному расстоянию (edit distance) как запасной вариант. Настроено на реальных ложных срабатываниях/пропусках из продакшена, а не только на happy-path случаях.
5. **Самовосстанавливающиеся CSS-селекторы с автослиянием** — когда скрейпер перестаёт возвращать концерты, автоматизированный процесс просит Gemini (структурированный вывод, ограниченный только полями селекторов — никогда не затрагивая резервные значения venue/city/country) исправить его, проверяет исправление на закэшированном HTML и полном наборе тестов, открывает PR для аудита изменений и **сразу же делает squash-merge**. Без участия человека.
6. **Несколько стратегий скрейпинга** — `static_selectors` и `json_api` покрывают большинство площадок; `custom_js`, `jsonld`, `next_data` и `playwright_render` (headless Chromium) обрабатывают остальные. Конфигурация `static_selectors` автоматически восстанавливается через JSON-LD, если её селекторы вдруг перестают находить хоть одно событие.
7. **Страховочная сетка деплоя** — health-gate отказывается публиковать данные при нулевом количестве событий или падении более чем на 50% относительно закэшированного базового уровня; SSRF-защита отклоняет конфигурации скрейперов, нацеленные на localhost/приватные/link-local/metadata-адреса; per-domain circuit breaker прекращает долбить сайт после 3 подряд неудачных попыток; freshness watchdog открывает GitHub issue, если запланированный запуск незаметно перестаёт срабатывать.
8. **Строгая проверка схемы** — каждый концерт перед публикацией проверяется на соответствие Zod-схеме, включая отбраковку календарно некорректных дат (например, перепутанные местами день/месяц) и отбрасывание концертов, дата которых уже прошла.

## Статические выходные данные API (`dist/`)

Развёрнуто по адресу `https://cartograf666.github.io/concert-for-travelers-api/`:

### Быстрый старт для потребителя

Чтобы получить массив предстоящих концертов, достаточно выполнить GET-запрос к `concerts.json`:

```javascript
fetch('https://cartograf666.github.io/concert-for-travelers-api/concerts.json')
  .then(response => response.json())
  .then(concerts => console.log(`Loaded ${concerts.length} concerts`));
```

Пример структуры JSON-объекта концерта:

```json
{
  "artist": "The Cure",
  "artistWebsite": "https://www.thecure.com/",
  "spotifyId": "7bu3v4mR1rR2880r3W6y23",
  "mbid": "cc197c19-090e-437c-9b41-f1b1967451c8",
  "artistSocials": {
    "spotify": "https://open.spotify.com/artist/7bu3v4mR1rR2880r3W6y23",
    "instagram": "https://www.instagram.com/thecure/",
    "facebook": "https://www.facebook.com/thecure",
    "youtube": "https://www.youtube.com/user/TheCureVideography",
    "telegram": null,
    "vk": null
  },
  "date": "2026-11-01",
  "startTime": "20:00",
  "venue": "Hanns-Martin-Schleyer-Halle",
  "city": "Stuttgart",
  "country": "DE",
  "lat": 48.7901,
  "lng": 9.2241,
  "ticketUrl": "https://www.thecure.com/",
  "originalSource": "ticketmaster.de",
  "scrapedAt": "2026-07-08T12:00:00.000Z"
}
```

- **`index.json`** — `schemaVersion` (увеличивается при изменении формы Concert, заметном для потребителя), метрики запуска (`lastRun`, `stats.totalConcerts/uniqueArtists/uniqueCities`), полный список уникальных артистов и полный список уникальных городов.
- **`concerts.json`** — полный master-массив всех предстоящих (никогда не прошедших по дате) концертов.
- **`artists.json`** — ПОЛНЫЙ справочник одобренных артистов (все ~63 000, а не только те, у кого сейчас есть концерт), с тем же ключом-slug, что и в `artists/{slug}.json`: `slug`, `name`, `website?`, `socials?`, `spotifyId?`, `mbid?`, `genres?`, `popularity?` (`listeners`/`playcount`), `image?`, `similarArtists?` (до 8 записей вида `{name, slug, match}`; `artist.getsimilar` из Last.fm, сверенный с тем же самым whitelist, так что каждая рекомендация указывает на реально существующий `artists/{slug}.json`).
- **`artists/{artist-slug}.json`** — концерты одного артиста (например, `artists/the-cure.json`), отсортированные по дате. **Примечание**: поскольку этот API статически хостится на GitHub Pages, для несуществующих файлов артистов будет возвращён HTML 404 вместо JSON. Массив `artists` в `index.json` содержит исходные отображаемые имена, а не slug'и — сначала примените `slugify()` к каждому имени, а затем сверяйте запрашиваемый slug с этим производным набором перед запросом (прямое сравнение исходного slug с массивом `artists` совпадения не даст).
- **`cities/{city-slug}.json`** — концерты одного города (например, `cities/berlin.json`), отсортированные по дате. Устаревшие пофайловые записи для артистов/городов, которые больше не гастролируют, вычищаются при каждом запуске. **Примечание**: то же самое, что и выше — примените `slugify()` к каждому имени в массиве `cities` из `index.json` перед сравнением.
- **`status.json`** — машиночитаемая сводка состояния (скрейперы ok/failed, устаревшие площадки, количество событий ticketmaster, `conflictDropsLast7Days` — см. [Известные ограничения](#known-limitations)) для дашборда и freshness watchdog.
- **`changes.json`** — концерты, появившиеся с момента последнего запуска (скользящее окно 30 дней), чтобы потребитель мог показать «N новых концертов с вашего последнего визита», не сравнивая весь `concerts.json` самостоятельно.

Каждый объект концерта соответствует `src/schemas/concert.ts`: `artist`, `artistWebsite?`, `spotifyId?`, `mbid?`, `artistSocials?` (spotify/instagram/facebook/youtube/telegram/vk), `date` (`YYYY-MM-DD`), `startTime?` (`HH:MM`), `venue`, `venueKind?` (stadium/arena/club/theatre/hall/open-air/other), `city`, `country` (ISO 3166-1 alpha-2), `lat?`/`lng?`, `festival?` (`{name, url?}`), `lineup?`, `priceRange?` (`{min, max, currency}`, только для Ticketmaster), `ticketUrl?` (собственный сайт артиста, если он известен, а не ссылка на продавца билетов — см. ниже), `originalSource`, `scrapedAt`.

**Примечание про `ticketUrl`**: несмотря на название, это собственный официальный сайт артиста, если он нам известен, а не страница покупки билетов — специфичные для источника ссылки на билеты/агрегаторы часто оказываются безымянным виджетом или обобщённой страницей со списком по городу без какого-либо указания на то, что это такое. Откат к исходной ссылке на билет из источника происходит только тогда, когда сайт артиста для него неизвестен.

## Локальная разработка и команды

### Предварительные требования
- Node.js v20+
- Ключ Gemini API (опционально, нужен для JIT-обогащения и самовосстановления)

### Установка
```bash
npm install
```

### Сборка кода
```bash
npm run build
```

### Запуск скрейпера и агрегатора
Запускает все скрейперы площадок + проход Ticketmaster, вливает всё, что в последний раз закэшировала задача artist-scrape, выполняет JIT-обогащение соцсетей/сайта, записывает результаты в `/dist` и логирует отчёты об ошибках в `reports/fail-log.json`.
```bash
# Опционально: задайте ключ Gemini для обогащения
export GEMINI_API_KEY="your-gemini-key"

npm run scrape
```

### Запуск прохода по тур-страницам артистов + Bandsintown + Eventbrite
```bash
npm run scrape-artists
```

### Запуск самовосстановления локально
Если у вас есть файл `reports/fail-log.json` и ключ Gemini API:
```bash
export GEMINI_API_KEY="your-gemini-key"
npm run heal
```

### Пополнение списка целевых артистов
Подтягивает актуальные чарты Deezer (без ключа) + Last.fm (опционально `LASTFM_API_KEY`) и добавляет ранее не встречавшихся популярных артистов в `data/artist_scrape_targets.txt`.
```bash
npm run discover-artists
```

Имя, добавленное в `data/artist_scrape_targets.txt`, должно также присутствовать в шардированной базе артистов в `data/artists/`, иначе его концерты, найденные скрейпингом/через Bandsintown/Eventbrite, будут отброшены как «не одобрено» -- идемпотентно, добавляет только то, чего не хватает.

### Геокодирование координат площадок
Разовый пакетный скрипт: геокодирует фиксированную площадку каждого скрейпера (название + город + страна) через бесплатный провайдер OpenStreetMap/Nominatim и кэширует полученные `lat`/`lng` в собственной JSON-конфигурации скрейпера. Пропускает скрейперы тур-страниц артистов (площадка задаётся построчно, единой фиксированной координаты нет) и конфигурации, у которых координаты уже есть.

## Процесс обогащения и самовосстановления — подробности

1. **Daily Scrape (ежедневный скрейпинг)**: запускает все скрейперы площадок + проход Ticketmaster, вливает последние закэшированные результаты задачи artist-scrape (тур-страницы + Bandsintown + Eventbrite), применяет сопоставление артистов + парсинг дат + нормализацию + фильтрацию прошедших дат, и пропускает результат через health-gate перед публикацией в `/dist` и деплоем на GitHub Pages. Для скрейпера, завершившегося с ошибкой или пустым результатом, образец HTML и ошибка логируются в `reports/fail-log.json`.
2. **Обогащение метаданных «сначала бесплатное»**: новые артисты обогащаются в порядке возрастания стоимости — сначала массовый проход Wikidata SPARQL (быстрый, ~80 имён за запрос), затем поартистный проход MusicBrainz + Wikidata каждые 3 часа, и только оставшийся длинный хвост отправляется в Gemini пакетами за website + соцсетями.
3. **Отказоустойчивое переключение между ключами и моделями**: и обогащение, и самовосстановление перебирают все настроенные ключи Gemini и каскад моделей Gemma/Gemini-Flash, переходя к следующему ключу только после исчерпания квоты/авторизации у всех моделей на текущем.
4. **Исправление селекторов через LLM**: после завершения Daily Scrape пайплайн самовосстановления скачивает артефакт fail-log и для каждой ошибки просит Gemini (структурированный вывод, только поля селекторов) проанализировать сломанные селекторы на закэшированном HTML.
5. **Самокоррекция с автослиянием**: исправленные селекторы проверяются на закэшированном образце HTML и полном наборе тестов. Если проверка пройдена, конфигурация площадки обновляется в новой ветке, открывается как PR (для аудита изменений) и **сразу же выполняется squash-merge** — без ручного этапа ревью.
6. **Freshness Watchdog**: независимо проверяет, что успешный ежедневный запуск произошёл недавно, и открывает дедуплицированный GitHub issue, если расписание незаметно перестало срабатывать, запуск завис или был пропущен.
7. **Безопасность при параллельной записи**: `main` получает коммиты от нескольких независимых задач (собственный коммит обогащения от daily-scrape, оба воркфлоу enrich-*, автослияние self-heal). Каждый шаг автокоммита при отклонении push повторяет попытку с fetch+rebase вместо провала всей задачи.

## Известные ограничения

- **У шардированной базы артистов в `data/artists/` несколько писателей**, координируемых через git-push-retry, а не настоящей транзакцией. Хотя шардирование предотвращает конфликты файлов для писателей, редактирующих разные шарды, конфликт на одном и том же шарде всё ещё возможен. `enrich-auto`/`enrich-database`/`daily-scrape` используют общую concurrency-группу `artist-db-write` (GitHub ставит их в очередь, так что буквально параллельно они не выполняются), но неразрешимый конфликт rebase с каким-то *другим* push в `main` иногда всё же заставляет писателя отбросить собственный коммит вместо провала задачи. Данные при этом не теряются (затронутые артисты просто остаются в ожидании и повторяются на следующем запуске), но это впустую потраченные API-вызовы/вычисления за ту попытку. Отслеживается в `data/conflict-drops.json` (через `npm run record-conflict-drop`) и выводится как `conflictDropsLast7Days` в `status.json` / на дашборде — следите за растущим счётчиком.
- **Нет жёсткого барьера совместимости схем.** `schemaVersion` в `index.json` — это сигнал («что-то изменилось в форме Concert, проверьте `src/schemas/concert.ts`»), а не принудительно соблюдаемый контракт — все изменения до сих пор были аддитивными, так что старый потребитель, игнорирующий неизвестные поля, в любом случае не пострадает.
