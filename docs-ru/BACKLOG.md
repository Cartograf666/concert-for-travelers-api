# Бэклог

Живой бэклог concert-data API. Репозиторий только для бэкенда/данных —
отдельное приложение (`concerts-for-travelers`) потребляет опубликованный JSON.

**Ключевой (north-star) пользовательский сценарий (потребительское приложение):**
Пользователь добавляет любимых артистов → указывает даты отпуска → приложение
возвращает, на какие концерты он может попасть и где. Задача этого репозитория —
питать этот сценарий данными, достаточно богатыми, чтобы (1) надёжно
**сопоставлять** артистов пользователя, (2) **размещать** каждый концерт в
пространстве и времени и (3) **ранжировать** варианты.

Легенда: ✅ готово · 🚧 в процессе · ⬜ запланировано · 💡 идея

## ✅ Готово

- **Источники концертов**: скрейперы площадок (91 конфиг), Ticketmaster (27 стран),
  Bandsintown (по артистам, по всему миру — покрывает Азию/Японию и артистов из RU),
  Eventbrite (по артистам, скрейпинг discovery-страниц US).
- **Статический JSON API на GitHub Pages**: `concerts.json`, `artists/{slug}.json`,
  `cities/{slug}.json`, `index.json`, `status.json`, дашборд `index.html`.
- **Данные по каждому концерту**: artist, artistWebsite, artistSocials (spotify/instagram/
  facebook/youtube/telegram/vk), date (YYYY-MM-DD), venue, city, country (ISO2),
  lat/lng (опционально), ticketUrl, originalSource, scrapedAt.
- **Самовосстанавливающиеся скрейперы**: сломанные CSS-селекторы автоматически
  чинятся через Gemini, валидируются, оформляются в PR и squash-мёрджатся без
  участия человека. Быстрый путь через JSON-LD применяется прежде, чем вызывать LLM.
- **База белого списка артистов**: ~63.5k артистов, многоуровневое сопоставление
  имён (точное → фильтр кавер/трибьют-групп → подстрока с защитой по покрытию →
  нечёткое). Соцсети есть у ~27k, обогащены через MusicBrainz (2.7k) + Wikidata-bulk
  (23.6k).
- **Самопополняющийся список целей**: `discover-artists` (еженедельно) забирает
  живые чарты → Deezer + Last.fm (+738 по всему миру). Список целей вырос с 1424
  до 2204. Spotify исключён (бесплатный тариф блокирует Web API).
- **Мульти-ключевой failover для Gemini**: переключение на следующий ключ
  (`GEMINI_API_KEY`, `_RESERV1/2`, `_2.._10`, `GEMINI_API_KEYS`), как только у
  всех моделей на текущем ключе исчерпана квота/авторизация. Покрывает enrich +
  self-heal + run. → `src/engine/gemini_keys.ts`.
- **Канонические ID артиста на каждом концерте: `spotifyId` + `mbid`.** `spotifyId`
  парсится из `artistSocials.spotify` (без вызова Spotify API); `mbid` берётся из
  MusicBrainz (`enrich_auto.ts`) и утверждения P434 в Wikidata (как в поартистном
  проходе, так и в массовом проходе SPARQL), плюс разовый `backfill_mbid.ts`,
  который задним числом проставляет `mbid` артистам, уже обогащённым до появления
  этого поля. → `src/schemas/concert.ts`, `src/pipeline/process.ts`,
  `src/scripts/enrich_auto.ts`, `src/scripts/enrich_wikidata_bulk.ts`,
  `src/scripts/backfill_mbid.ts`.
- **Гарантированное геокодирование.** `src/pipeline/geocode.ts`: заполняет
  lat/lng для любого концерта, у которого их нет (в основном площадки со
  страниц туров отдельных артистов), через персистентный кэш с ключом
  venue+city+country, так что повторная площадка обходится в один запрос к
  Nominatim за всё время, с лимитом за прогон и переносом остатка на следующий
  прогон. Намеренно вынесено за пределы `processConcerts` (который остаётся
  сетенезависимым и юнит-тестируемым) — подключено в `run.ts` как отдельный шаг
  best-effort. Поле схемы остаётся опциональным по замыслу: жёсткое требование
  означало бы, что одно неразрешимое название площадки ломает весь публикейт.
  → `src/pipeline/geocode.ts`, `src/run.ts`.
- **Жанры + популярность + изображение артиста.** `src/scripts/enrich_metadata.ts`:
  один вызов Last.fm `artist.getInfo` на артиста даёт как топ-теги (жанры), так
  и статистику слушателей/прослушиваний (нужен бесплатный `LASTFM_API_KEY`, при
  его отсутствии шаг аккуратно пропускается — та же конвенция, что и в
  `discover_artists.ts`); поиск артиста в Deezer (без ключа) даёт изображение,
  которому доверяют только при точном совпадении нормализованного имени.
  Собственный gate ожидающих (`metaEnrichedAt`/`metaTriedAt`), независимый от
  уровней обогащения идентичности, поскольку у большинства белого списка там
  уже стоит `enrichedAt`. *Сокращение объёма:* фоллбэк на изображение
  Ticketmaster-attraction был пропущен — поиск артиста в Deezer без ключа уже
  покрывает подавляющее большинство реально гастролирующих артистов, а
  подключение изображения TM по каждому событию к записи в БД изнутри
  ежедневного прогона скрейпинга добавило бы реальную связанность ради
  небольшого прироста покрытия. Запускается по расписанию через
  `.github/workflows/enrich-metadata.yml` (cron, с чекпоинтами, той же формы,
  что и `enrich-auto.yml`), также можно запустить вручную через
  `npm run enrich-metadata [N]`. → `src/scripts/enrich_metadata.ts`.
- **Полный каталог артистов `dist/artists.json`.** Публикует *весь* белый
  список (а не только артистов с текущим концертом), с тем же slug-ключом, что
  используют файлы концертов по артистам, с name/website/socials/spotifyId/mbid/
  genres/popularity/image, когда они известны. Закрывает пробел, из-за которого
  `index.json` перечислял только артистов, у которых уже был отскрейпленный
  концерт, так что автодополнение «добавить любимых артистов» в потребительском
  приложении никак не могло увидеть остальные ~60k артистов из белого списка. →
  `src/generator/publish.ts` (`publishArtistCatalog`), подключено в `src/run.ts`.
- **`startTime` события.** HH:MM, заполняется из `localTime` Ticketmaster,
  ISO-даты Bandsintown и любого источника, чья строка даты содержит ISO-время
  (например, `startDate` из JSON-LD). Намеренно *не* добавляет второй проход
  chrono-node по датам скрейперов в свободном тексте (соотношение затрат и
  выгоды) — см. docstring у `extractTimeFromRawDate`. → `src/schemas/concert.ts`,
  `src/pipeline/process.ts`, `src/engine/ticketmaster.ts`, `src/engine/bandsintown.ts`.
- **Тип площадки (venue kind).** Классификатор на основе ключевых слов
  (`stadium`/`arena`/`club`/`theatre`/`hall`/`open-air`/`other`), применяется к
  названию площадки каждого концерта единообразно для всех источников, вместо
  особых случаев для каждого источника отдельно. → `src/pipeline/process.ts`
  (`inferVenueKind`).
- **Учёт фестивалей.** События Ticketmaster с более чем одной записью в
  `_embedded.attractions` трактуются как афиша с несколькими артистами:
  `festival {name, url}` + `lineup[]` (остальные артисты в афише). *Сокращение
  объёма:* определение фестивалей для скрейперов площадок не пробовали делать —
  для этого нет общего сигнала на отскрейпленной странице, в отличие от массива
  attractions, который TM даёт бесплатно; потребовались бы доработки в
  конфигурации каждого скрейпера отдельно, что является отдельной и намного
  большей работой. → `src/schemas/concert.ts`, `src/engine/ticketmaster.ts`,
  `src/pipeline/process.ts`.
- **Unicode-фикс в `slugify()`.** Ранее существовавший баг, выявленный при
  нагрузочном тестировании нового каталога `dist/artists.json` на реальном
  белом списке в полном масштабе: имя не только на латинице (кириллица, CJK,
  ...) обрезалось до пустой строки ASCII-only фильтром `\w`, из-за чего
  91 из 63 490 артистов схлопывались в один файл/slug (374 отдельные группы
  коллизий, 758 имён, если считать и частичное искажение тоже). Теперь
  Unicode-aware (`\p{L}`/`\p{N}`) со стабильным фоллбэком на хэш для имён
  вообще без букв/цифр. → `src/pipeline/process.ts` (`slugify`).
- **Лента изменений `dist/changes.json`.** Концерты, новые с момента
  последнего прогона, чтобы потребитель мог показать «N новых концертов с
  вашего последнего визита», не сравнивая весь `concerts.json` самостоятельно.
  Идентичность переиспользует собственный ключ дедупликации
  `processConcerts` (artist+date+city) — а не второе определение «того же
  концерта». Состояние (какие концерты уже были известны) намеренно *не*
  отслеживается в git — сохраняется через тот же механизм `actions/cache`,
  который уже использует `reports/scrape-cache.json`, чтобы не добавлять ещё
  одного писателя в конкуренцию за `data/approved_artists.json`. Холодный
  старт (самый первый прогон) сообщает ноль изменений, а не все концерты
  разом. Окно хранения — 30 дней. → `src/generator/changelog.ts`, подключено
  в `src/run.ts`.
- **`priceRange`.** Цена билета best-effort (`{min, max, currency}`),
  только из собственного структурированного `priceRanges` Ticketmaster —
  схлопывает несколько уровней (например, standard + VIP) в общий min/max.
  Никогда не угадывается/не парсится из отскрейпленного свободного текста;
  скрейперы площадок это поле просто не получают. → `src/schemas/concert.ts`,
  `src/engine/ticketmaster.ts`, `src/pipeline/process.ts`.
- **Шардированное хранение белого списка артистов
  (`data/artists/shard-0.json`..`shard-7.json`, вместо единого 17-мегабайтного
  `data/approved_artists.json`).** Первопричина реального, неоднократно
  наблюдавшегося сбоя: каждый воркфлоу обогащения (enrich-auto,
  enrich-database/wd-bulk, enrich-metadata, собственный шаг enrich ежедневного
  скрейпинга) читает и перезаписывает ВЕСЬ файл артистов, поэтому два писателя,
  оказавшиеся рядом по времени, соревновались за пуш, и работа одного из них
  терялась на неразрешимом конфликте rebase — это уже сериализовано через
  `concurrency: artist-db-write`, но это предотвращает только параллельные
  прогоны, а не прогоны подряд, достаточно близкие друг к другу, чтобы всё
  равно столкнуться на одном и том же гигантском файле. Шардирование по первому
  символу имени артиста (mod 8) означает, что два писателя реально конфликтуют,
  только если затронули *один и тот же* шард — в большинстве случаев это не
  так, так что большинство конфликтов теперь структурно невозможны, а не просто
  повторяются в надежде на разрешение. Новый `src/pipeline/artistDb.ts`
  централизует все загрузки/сохранения за `loadApprovedArtists()`/
  `saveApprovedArtists()` (двойной режим: путь с суффиксом `.json` трактуется
  как устаревший однофайловый формат, который всё ещё используют временные
  фикстуры тестового набора; любой другой путь трактуется как шардированная
  продакшн-директория), с diff-перед-записью на каждый шард, чтобы сохранение,
  затронувшее лишь несколько артистов, не перезаписывало заодно файлы всех
  остальных шардов. Все 13 точек вызова, которые раньше читали/писали
  `data/approved_artists.json` напрямую (`pipeline/process.ts`,
  `pipeline/enrich.ts`, `run.ts` и 10 `scripts/*.ts`), теперь проходят через
  этот модуль. → `src/pipeline/artistDb.ts`, `data/artists/`.
- **Eventbrite как 4-й источник концертов (по артистам).** Eventbrite
  отключил свой публичный API поиска событий по нескольким организаторам для
  третьих сторон в декабре 2019 (API v3 покрывает только события, для которых
  вы уже знаете id/площадку/организацию) — подтверждено вживую, никакого более
  нового публичного продукта для поиска не существует. Единственный оставшийся
  путь — скрейпинг публичных discovery-страниц `/d/<location>/<query>/`,
  которые встраивают JSON-блоб `window.__SERVER_DATA__` с теми же результатами,
  что рендерит страница. **Это прямо нарушает условия использования (Terms of
  Service) Eventbrite** (раздел 13.1 запрещает скрейпинг) — осознанный,
  принятый риск (та же правовая категория, что и у любого скрейпера сайта
  площадки здесь, но против платформы с явным, заметным пунктом против
  скрейпинга), поэтому намеренно ведётся в малом объёме/вежливо (интервал 2.5с,
  лимит 300 артистов за прогон — меньше и мягче, чем у Bandsintown). Подтверждено
  вживую, что поиск Eventbrite `/d/` — это полнотекстовый поиск по ВСЕМУ
  каталогу, а не настоящий поартистный запрос, как у эндпоинта Bandsintown —
  например, запрос «Dropkick Murphys» выдал мастер-классы по уходу за волосами
  и гольф-туры, которые просто содержат слово «Murphy», а каждый результат
  первой страницы по «Metallica» оказался трибьют-группой. `mapEbResultToConcert`
  требует, чтобы запрошенное имя артиста стояло В НАЧАЛЕ заголовка результата,
  как предварительный фильтр релевантности (жертвует частью полноты ради
  заметно меньшего числа ложных срабатываний), прежде чем это вообще увидит
  общий фильтр кавер/трибьют-групп (`process.ts`). Ограничено локацией
  `united-states` (в discovery UI Eventbrite нет поиска «везде» — сокращение
  объёма, а не полное покрытие; переопределяется через
  `EVENTBRITE_LOCATION_SLUG`). Та же форма батчей/возобновляемости/фоллбэка на
  кэш, что и у обхода Bandsintown, с общим списком целей
  `data/artist_scrape_targets.txt`. → `src/engine/eventbrite.ts`,
  `src/run-artists.ts`, `src/run.ts`, `.github/workflows/artist-scrape.yml`,
  `.github/workflows/daily-scrape.yml`.
- **Рекомендации «похожие артисты».** Один вызов Last.fm
  `artist.getsimilar` на артиста, сверяется с нашим собственным белым списком
  ~63k, так что каждая рекомендация разрешается в реальный `artists/{slug}.json`
  — рекомендация, указывающая за пределы этого каталога, была бы тупиком для
  потребительского приложения, а не фичей. До 8 записей `{name, slug, match}`
  на артиста, с сохранением убывающего по match порядка Last.fm. Собственный
  gate ожидающих (`similarEnrichedAt`/`similarTriedAt`), независимый от всех
  остальных уровней обогащения, та же логика, что и у уровня жанров/
  популярности. Запускается по расписанию через
  `.github/workflows/enrich-similar.yml` (cron, с чекпоинтами, той же формы,
  что и `enrich-metadata.yml`), также можно запустить вручную через
  `npm run enrich-similar [N]`. → `src/scripts/enrich_similar_artists.ts`,
  `src/generator/publish.ts` (`ArtistCatalogEntry.similarArtists`).
- **Фильтр спам-площадок Bandsintown.** Расследование вживую подтвердило
  сфабрикованные RU-даты туров для реальных артистов (например, «Сплин в
  Ижевске») — публичная лента виджета Bandsintown позволяет третьим сторонам
  прикреплять события к любой странице артиста без какой-либо проверки.
  `isTemplatedArtistCityVenueName()` отклоняет любое событие, чьё название
  площадки соответствует спам-паттерну `"<artist> in/в <city>"`. →
  `src/engine/bandsintown.ts`.
- **`ticketUrl` теперь предпочитает собственный сайт артиста прямой ссылке
  на продавца билетов.** Продуктовое решение: ссылка на покупку билета часто
  сбивает с толку вне контекста (непонятно, что вообще это за страница);
  известный собственный сайт артиста — более безопасная страница-приземление
  по умолчанию. Откатывается на исходную ссылку на билет из источника только
  когда сайт артиста неизвестен. → `src/pipeline/process.ts`.
- **Автоматическая обрезка (prune) окончательно мёртвых конфигов
  скрейперов.** `heal.ts` намеренно пропускает ошибки
  `fetch_error`/`csr_detected`/`circuit_open` (сломанный CSS-селектор нельзя
  починить на странице, которая вообще не загрузилась) — такие конфиги
  накапливались бесконечно без какой-либо автоматической очистки.
  `prune_dead_scrapers.ts` отслеживает серии подряд идущих сбоев по каждому
  скрейперу (`data/scraper-health.json`), и как только конфиг набирает 5
  подряд идущих «обрезаемых» сбоев, удаляет мёртвый конфиг скрейпера + сбрасывает
  у этого артиста маркеры `tourUrl`/`tourScraperTriedAt`/`tourScraperCreatedAt`/
  `tourUrlProbeTriedAt`, чтобы он снова стал пригоден для повторного обнаружения,
  логируя каждую обрезку в `data/pruned-scrapers.json` для аудита. Пропускает
  сброс полей артиста (но всё равно обрезает мёртвый скрейпер), когда 2+
  записи в БД имеют совпадающее без учёта регистра имя, вместо того чтобы
  гадать, какую из них трогать. Запускается после каждого ежедневного
  скрейпинга через `workflow_run`, входит в группу конкурентности
  `artist-db-write`. → `src/scripts/prune_dead_scrapers.ts`,
  `.github/workflows/prune-dead-scrapers.yml`.
- **Фоллбэк на LLM-извлечение при прогонах скрейпера с нулевым
  результатом.** Закрывает разрыв между поломкой CSS-селектора и его починкой
  через `heal.ts`: когда и статический селектор, и существующий бесплатный
  фоллбэк на JSON-LD возвращают ноль событий на странице, которая нормально
  загрузилась и не является CSR, просим Gemini извлечь концерты напрямую из
  того же уже загруженного HTML. Бюджет на прогон (30 вызовов,
  race-safe синхронная проверка-затем-декремент), вывод `ticketUrl` проходит
  через `safeAbsoluteUrl()`, как и у любого другого источника, риск
  галлюцинаций ограничен уже существующей проверкой по белому списку

## 🚧 В процессе

### Пробел в покрытии Bandsintown (клиент замерил потерю ~60% по сравнению с прямым live-запросом)
Собственный compare-скрипт потребительского приложения замерил
концерты, присутствующие в выходных данных публикации этого репозитория, но
отсутствующие, когда оно само запрашивает Bandsintown вживую — ~60% для
выборочно проверенных артистов (AC/DC, A Day To Remember — оба уже в
`data/artist_scrape_targets.txt`). Первопричина подтверждена именно в этом
репозитории, а не на стороне клиента: обход Bandsintown в `artist-scrape.yml`
запускался **еженедельно** по 800/2205 целей за прогон (~3-недельный полный
цикл) — артист мог оставаться неактуальным неделями до следующего запроса.

План (последовательный; не трогать клиент/не отключать сервер, пока не
пройден финальный gate):
1. ✅ Cron `weekly -> daily` (`.github/workflows/artist-scrape.yml`).
   2205/800 ≈ 3 дня на первичное заполнение бэклога; устоявшаяся
   пропускная способность (~367/день для 6-дневного окна свежести,
   `DEFAULT_FRESHNESS_DAYS` в `src/engine/bandsintown.ts`) с комфортным
   запасом укладывается в лимит 800/день — ежедневного запуска достаточно, без
   избытка. Репозиторий публичный, так что минуты GH Actions бесплатны;
   вопрос стоимости не стоит.
2. ✅ Дедупликация дублей-вариантов регистра в
   `data/artist_scrape_targets.txt`: 2205 → 2126 (удалены 78 дублей-вариантов
   регистра, например «A Day to Remember»/«A Day To Remember»/
   «a day to remember» считались тремя отдельными слотами запросов к
   Bandsintown для одного реального артиста, поскольку собственная
   дедупликация в `fetchBandsintownConcerts` чувствительна к регистру). →
   `src/scripts/clean_scrape_targets.ts`, запущено на живом списке целей.
3. ✅ Направить скрейперы официальных страниц туров
   (`scrapers/artists/*.json` — надёжнее публичной ленты виджета Bandsintown,
   без риска rate-limit/блокировки) на наши самые ценные цели. Намеренно *не*
   на основе данных использования самого потребительского приложения (список
   сохранённых избранных пользователя — смещённый, вручную поддерживаемый
   прокси) — вместо этого ранжировано по реальной популярности в Last.fm
   (`entry.popularity.listeners`, собирается `enrich_metadata.ts`, с живым
   запросом к Last.fm как фоллбэком) среди артистов, уже находящихся в нашем
   собственном списке целей. `npm run rank-scraper-candidates [N]` / воркфлоу
   `rank-scraper-candidates.yml` с ручным запуском (нужен `LASTFM_API_KEY`,
   уже есть как секрет репозитория) выдаёт ранжированный список; для того
   чтобы по нему реально написать конфиг селекторов для каждого артиста,
   всё ещё нужен человек. → `src/scripts/rank_scraper_candidates.ts`.
4. ⬜ **Нужно реальное прошедшее время.** Дать новому ежедневному cron
   отработать 1-2 реальных цикла (как минимум одно 6-дневное окно свежести),
   прежде чем перемерять.
5. ⬜ **Нужны инструменты клиентского репозитория.** Перезапустить
   compare-скрипт потребительского приложения на свежих данных.
6. ⬜ Решающий gate: разрыв снизился до ~10-15% или меньше → отключить
   сервер, клиент переходит на обычный статический fetch. Всё ещё высокий →
   проблема в покрытии (нехватка скрейперов/целей), а не в частоте cron —
   расширять шаг 3, а не перенастраивать cron.
